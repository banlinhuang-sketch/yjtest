# `project-todo-plan-2026-03-18` 分阶段实施方案

## Summary

当前目标不是继续扩页面，而是把现有“联调版”收成“可正式接后端、可稳定验收、可持续维护”的交付版本。

默认技术路线：

- 前端继续使用当前 React 结构，不引入新路由框架或新状态库
- API 继续走 `REST JSON + Bearer Token + sessionStorage`
- 现有 `TestCase` 前端模型继续保留，通过 DTO adapter 适配后端
- `server/` 目录作为开发联调后端，最终会被正式业务后端替换或降级为 mock

## 阶段划分

### 阶段 1：冻结接口契约，完成正式后端接入准备

目标：把“本地联调后端可用”推进到“正式后端可无缝替换”。

关键内容：

- 固化认证、用例、审核、导出四类接口契约
- 明确 `CaseSummaryDTO`、`CaseDetailDTO`、导出任务 DTO、认证用户 DTO
- 明确错误语义：`401 / 403 / 422 / 5xx / network`
- 明确 token 生命周期和前端失效处理策略

完成定义：

- 所有一期接口都有稳定契约和示例
- 前端不再依赖“猜测字段名”或临时 adapter 修补
- 正式后端团队可以按契约直接实现或对接

### 阶段 2：切换正式认证与用例接口

目标：先打通最核心链路，完成 `P0 / P1 / P2` 的正式后端切换。

关键内容：

- `P0` 登录页从本地 Node API 切换到正式认证接口
- `App.tsx` 启动时用正式 `auth/me` 恢复登录态
- `P1` 工作台真实拉取列表、真实筛选、真实生成草案
- `P2` 保存改为正式 `PATCH /cases/:id`

完成定义：

- 正式登录成功/失败可用
- `P1` 首屏能拉到真实列表
- `P1` 生成草案可真实新增并自动进入编辑
- `P2` 保存后重新进入页面仍能读到同一条真实数据

### 阶段 3：切换正式审核与导出接口

目标：完成 `P3 / P4` 的正式后端切换，打通完整业务链路。

关键内容：

- `P3` 审核动作接正式 `POST /cases/:id/review`
- `待审核 / 已处理` 视图都使用真实列表数据
- `P4` 改为正式“创建任务 -> 轮询任务 -> 下载文件”
- 导出失败态与错误文案由后端真实状态驱动

完成定义：

- 审核通过 / 退回能真实更新状态
- 用例能在 `待审核 / 已处理` 间正确流转
- 导出任务能从创建走到成功或失败

### 阶段 4：补齐 P5 / P6 的真实化与错误承接

目标：把当前“主要是展示”的页面补成正式系统的一部分。

关键内容：

- `P5` 知识基线页接真实只读接口
- `P6` 细化成正式错误码与异常场景映射页
- 页面级错误基于正式返回结果跳转或渲染

完成定义：

- `P5` 不再依赖本地 mock 数据
- `P6` 的状态来源清晰且与正式接口一致
- 页面级异常有统一 fallback

### 阶段 5：回归、自动化与发布前收口

目标：把“能跑”升级为“可稳定回归、可交付”。

关键内容：

- 形成固定浏览器回归清单，覆盖 `P0 -> P1 -> P2 -> P3 -> P4`
- 修复或确认 `P1 生成草案` 在自动化点击中的稳定性
- 补项目文档与联调说明
- 明确本地 `server/` 是开发 mock 还是保留调试后端

完成定义：

- `npm.cmd run lint` 通过
- `npm.cmd run build` 通过
- 浏览器主链路全绿
- 有一份可交付的联调 / 回归说明文档

## Important Interfaces

- 认证
  - `login(username, password)`
  - `getCurrentUser()`
- 用例
  - `listCases(filters)`
  - `getCaseDetail(id)`
  - `createCase(payload)`
  - `updateCase(id, payload)`
  - `generateDraft({ scope, title, requirement })`
  - `reviewCase(id, { action, reviewNote })`
- 导出
  - `createExportTask({ format, filters })`
  - `getExportTask(taskId)`
- 知识基线
  - `listKnowledgeSources()`

## Test Plan

1. 登录成功、登录失败、刷新恢复登录态、token 失效回到登录页
2. `P1` 列表加载、搜索、范围筛选、优先级筛选、生成草案、克隆保存
3. `P2` 详情读取、保存成功、保存失败、离开拦截
4. `P3` 待审核 / 已处理切换、审核通过、退回草稿、审核意见必填
5. `P4` 导出任务创建、轮询、成功下载、失败重试
6. `P5` 知识源读取与空状态
7. `P6` 登录失效、无权限、网络异常、系统异常映射

工程回归：

- `npm.cmd run lint`
- `npm.cmd run build`
- 浏览器回归至少覆盖 `P0 -> P1 -> P2 -> P3 -> P4`

## Assumptions

- 当前真正未完成的重点不是页面，而是“从本地联调后端切到正式后端”
- 一期仍不做 refresh token、SSO、权限中心、导出历史
- `P5` 只做只读真实化，不扩展编辑后台
- 本地 `server/` 最终定位为开发 mock / fallback，不作为正式交付后端
- 如资源有限，优先级顺序固定为：
  - 正式认证与 cases
  - 正式 review 与 exports
  - P5 / P6 真实化
  - 自动化与文档收口
